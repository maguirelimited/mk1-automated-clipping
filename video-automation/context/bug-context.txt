1. select_clip.py gives the model no real timestamp map

Problem:
select_clip.py loads only data["text"] from the Whisper transcript. It does not give the model Whisper segment timestamps. So the model is asked to return timestamps, but it only sees plain transcript text. That is why it can invent 00:15:00 even though the video is only 00:09:07.

Wrong location:
_load_transcript() only returns:

data["text"]

Outcome needed:
The selector should use Whisper segments, not just raw text, so timestamps are grounded in real transcript timings.

2. select_clip.py prompt says “timestamps in the transcript” but the transcript has no timestamps

Problem:
The prompt says:

start and end must be timestamps in the transcript

But the transcript inserted into the prompt is just plain text. So this instruction is impossible to follow accurately.

Outcome needed:
Either include timestamped transcript chunks in the prompt, or stop asking the model to invent exact timestamps from untimed text.

3. pipeline_utils.py validates duration, but not against actual video length

Problem:
postprocess_segments() checks:

start < end
duration between min and max
overlap / duplicate rules

But it never checks:

end <= video_duration
start < video_duration

So 00:15:00 → 00:15:30 passes, because it is a valid 30-second segment mathematically.

Outcome needed:
Add video-duration bounds checking before clips are accepted.

4. app.py has _ffprobe_duration_sec() but doesn’t use it to validate selected segments

Problem:
app.py can already get video duration using _ffprobe_duration_sec(path), but inside _run_pipeline() it only uses it later for debug logging after clipping. By then it is too late.

Wrong location:
This validation should happen before:

for index, segment in enumerate(processed_segments, start=1):

Outcome needed:
Run ffprobe before clipping, then discard or fail any segment where start >= video_duration or end > video_duration.

5. app.py post-selection correction only uses Whisper segments if it can find matching times

Problem:
When the selected start is 00:15:00, _find_segment_for_time() cannot find a Whisper segment because the transcript ends around 9 minutes. But _correct_segment_start() just returns the original bad segment when no matching segment is found. Same for _correct_segment_end().

Wrong behavior:
Invalid timestamp → no Whisper segment found → bad segment is kept.

Outcome needed:
Invalid timestamp → segment should be rejected, not preserved.

6. clip_video.py accepts out-of-range timestamps because it only checks start < end

Problem:
clip_video.py calls:

validate_segment_times(start, end)

That only confirms the timestamps parse and start is before end. It does not know the input video duration.

Outcome needed:
Either clip_video.py should ffprobe the video duration itself, or app.py should guarantee only valid timestamps reach it. Best is both.

7. clip_video.py treats tiny/invalid ffmpeg output as valid if file size is not zero

Problem:
The broken clip had:

video:0KiB audio:1KiB
Lsize=3KiB

Your script only checks:

os.path.getsize(output_video) == 0

So a 3KB garbage clip passes as “valid”.

Outcome needed:
After ffmpeg, use ffprobe to check the output duration. If the output clip duration is near 0 or far below expected duration, fail it.

Short version

The real chain is:

select_clip.py gives untimed transcript → model invents 15:00 timestamp → pipeline_utils accepts it because duration is 30s → app.py does not reject out-of-bounds times → clip_video.py cuts beyond video length → tiny blank clip passes because file size > 0

Main fixes:

1. Give selector timestamped Whisper segments.
2. Validate selected timestamps against video duration before clipping.
3. Reject segments that do not map to Whisper segments.
4. Validate final clip duration after ffmpeg.

Update the pipeline to enforce strict timestamp validity and grounding. The core issue is that the selector currently operates on untimed transcript text, which causes the model to invent timestamps that exceed the actual video duration. 
First, modify select_clip.py so that the model receives Whisper segment data (start, end, text) instead of raw transcript text, formatted in a readable timestamped structure, ensuring all selections are grounded in real timings. 
Second, in app.py, immediately after retrieving video_duration_sec using ffprobe, add a hard validation step that filters out any segments where start >= video_duration or end > video_duration, instead of just logging them . 
Third, update _correct_segment_start and _correct_segment_end so that if a timestamp cannot be mapped to a Whisper segment, the segment is rejected rather than returned unchanged.
Fourth, ensure that only validated segments reach clip_video.py, but also add a secondary safety check there by comparing expected duration vs actual ffprobe duration of the output clip and rejecting clips that are near-zero or significantly shorter than expected, since file size alone is not a valid integrity check.
The goal is to eliminate all invalid timestamps at the source, enforce bounds before clipping, and guarantee that every output clip corresponds to a real, meaningful segment of the original video.
